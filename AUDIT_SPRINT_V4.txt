── 6. SERVER ACTIONS PATTERN ──────────────────────────────

Le projet utilise EXCLUSIVEMENT les Server Actions Next.js ('use server').
Aucune route /api/ n'est utilisée pour l'admin (sauf /api/contact et /api/devis
qui sont des Route Handlers publics).

Pattern d'un actions.ts admin :
  - 'use server' en tête
  - import { revalidatePath } from 'next/cache'
  - import { createClient } from '@/lib/supabase/server'
  - Types exportés (utilisés par les Client Components)
  - Fonctions async exportées directement (pas de wrapper)
  - revalidatePath() à la fin des mutations

Exemple complet deletePhoto() :
  export async function deletePhoto(id, storagePath): Promise<{ error?: string }> {
    const supabase = await createClient()
    const { error: storageError } = await supabase.storage.from('photos').remove([storagePath])
    if (storageError) return { error: storageError.message }
    const { error: dbError } = await supabase.from('photos').delete().eq('id', id)
    if (dbError) return { error: dbError.message }
    revalidatePath('/admin/photos')
    return {}
  }

── 10. POINTS D'ATTENTION SPRINT V4 ───────────────────────

   La table n'a pas de contrainte UNIQUE sur is_featured.
   Pour garantir qu'il n'y a qu'une seule vidéo featured à la fois,
   il faut soit :
     Option 1 : deux UPDATE séparés dans la Server Action (risque de race condition minime)
     Option 2 : RPC Postgres (transaction atomique) — plus propre
